# SuperResVideoPlayer

A native macOS video player with real-time **Super Resolution** upscaling,
**AI Frame Interpolation** smoothing, and an **AI Subtitle Generator**, built
with SwiftUI, AVFoundation, Apple's MetalFX APIs (Spatial Scaler + Frame
Interpolator), and the Speech framework. Targets macOS 27 "Golden Gate" /
macOS 26 "Tahoe" — **macOS 26 is the hard minimum**, because Frame
Interpolation (`MTLFXFrameInterpolator`, part of Metal 4) isn't available
before it.

## How it works

- `PlayerViewModel` wraps `MPVPlayer` (`MPVPlayer.swift`) — a libmpv-backed
  engine, the same one IINA embeds — for playback/transport. libmpv demuxes
  and decodes every container/codec its bundled ffmpeg supports (MKV, WebM,
  FLAC, ...) natively, handles audio output and A/V sync internally, and
  hands decoded video frames to us via mpv's software render API as raw
  BGRA, Metal-compatible `CVPixelBuffer`s. Requires `brew install mpv`.
- `MetalVideoView` (an `NSViewRepresentable`) hosts an `MTKView` and hands frame
  rendering to `Renderer`.
- `Renderer` runs once per display refresh:
  1. Grabs the current video frame from the video output.
  2. Wraps it as a zero-copy `MTLTexture` via `CVMetalTextureCache`.
  3. If Super Resolution is enabled, feeds it through `MTLFXSpatialScaler`
     (Apple's MetalFX upscaler — the same technology used to boost game
     rendering) to produce a higher-resolution texture.
  4. Draws the resulting texture to the drawable with a minimal full-screen
     triangle shader (`Shaders.metal`).
- The UI (`ContentView`) exposes a Super Resolution toggle and scale factor
  picker (1.3x / 1.5x / 2.0x), an AI Frame Interpolation smoothing picker
  (Off / 2x / 3x), plus standard transport controls.

### AI Frame Interpolation

- `OpticalFlowEstimator` runs Vision's `VNGenerateOpticalFlowRequest`
  asynchronously (off the render thread) on each new pair of decoded frames,
  producing a per-pixel motion field.
- `Renderer` tracks the last two real decoded frames and, on every display
  refresh, computes how far playback has progressed between them (a
  continuous 0–1 fraction derived straight from `AVPlayerItemVideoOutput`'s
  own clock — no manual frame-rate bookkeeping needed). Depending on the
  chosen multiplier it either shows a real frame or synthesizes one:
  - **2x**: the one required in-between frame is generated by Apple's native
    `MTLFXFrameInterpolator` (`FrameInterpolator.swift`), fed the optical
    flow field as its motion input and a flat/constant texture as its depth
    input (video has no real depth — see the caveat below).
  - **3x**: `MTLFXFrameInterpolator` only produces a single fixed midpoint
    frame per Apple's documentation, so getting two evenly-spaced in-between
    frames instead uses a custom bidirectional motion-compensated warp+blend
    compute kernel (`warpBlendKernel` in `Shaders.metal`) driven by the same
    optical flow field. It's cruder than MetalFX's ML-based interpolation
    (no disocclusion/hole-filling) but works at an arbitrary temporal phase.
  - If Vision hasn't finished computing flow for the current pair yet, or
    `MTLFXFrameInterpolator` is unsupported on the GPU, the renderer falls
    back gracefully to showing the nearer real frame instead of stalling.
- Frame interpolation and Super Resolution compose: whatever frame gets
  chosen (real or synthesized) is then optionally upscaled by the spatial
  scaler before being drawn.

**Caveat — this is a game-engine API repurposed for video.**
`MTLFXFrameInterpolator` expects motion vectors and depth the way a 3D
renderer produces them. Video playback has neither natively: motion vectors
here come from Vision's optical flow estimate (which can be wrong around
fast/complex motion, transparency, or low-contrast regions) rather than
ground-truth per-pixel motion, and depth is a flat constant rather than real
scene geometry, so MetalFX can't use it to resolve disocclusion the way a
game would. Expect occasional ghosting/artifacts around fast-moving edges,
especially at 3x. This is a reasonable starting point, not a drop-in
replacement for dedicated video frame-interpolation models (e.g. RIFE).

**Verify against your SDK.** The property names in `FrameInterpolator.swift`
(`prevColorTexture`, `motionVectorScaleX/Y`, `depthTexture`, `deltaTime`,
`nearPlane`/`farPlane`/`fieldOfView`/`aspectRatio`, etc.) were checked against
an actual `MTLFXFrameInterpolator.h`/`MTLFXFrameInterpolatorDescriptor.h`
header diff for the Xcode 26 SDK found via web search — not from compiling on
this machine, since no macOS/Metal toolchain was available while writing
this. They should be accurate, but it's worth a quick diff against your local
`MetalFX.framework/Headers/` if anything doesn't compile.

**One more soft spot:** the motion texture comes from a `CVPixelBuffer` that
Vision's `VNGenerateOpticalFlowRequest` allocates internally — this project
doesn't control whether Apple marks that buffer Metal-compatible. If
`CVMetalTextureCacheCreateTextureFromImage` can't wrap it (see
`OpticalFlowEstimator.makeTexture`), `latestResult` just never gets set and
the renderer quietly falls back to showing real frames only — a safe failure
mode, but one to watch for if 2x/3x smoothing never seems to kick in.

### AI Subtitle Generator

- Click **Generate Subtitles** (after a video is loaded and its duration is
  known) to transcribe its audio with Apple's Speech framework
  (`SubtitleGenerator.swift`). It prefers **on-device** recognition when the
  selected language supports it (`SFSpeechRecognizer.supportsOnDeviceRecognition`),
  falling back to Apple's server-based recognition otherwise — the latter
  sends audio to Apple, which is disclosed in the Info.plist usage string.
- `SFSpeechURLRecognitionRequest` reads the video file directly — no manual
  audio extraction step is needed.
- Word-level segments (`SFTranscriptionSegment`, each with a timestamp and
  duration) get grouped into subtitle cues by a simple heuristic: start a new
  cue after a >0.35s pause, once a cue would exceed ~84 characters, or once
  it would span more than 6 seconds. This is a rough stand-in for a real
  sentence/clause segmenter, so cue breaks won't always land naturally.
- The **Language** picker lists every locale `SFSpeechRecognizer` supports on
  your Mac (`SubtitleGenerator.supportedLocales`), defaulting to your
  system's current locale if it's supported.
- Subtitles render as an overlay (`ContentView`'s `ZStack`) synced to
  `currentTime`, toggleable independently of generation. **Export .srt…**
  saves the current cues as a standard SubRip file via `SubtitleExporter`.
- The first time you click **Generate Subtitles**, macOS will show a Speech
  Recognition permission prompt (using the description in `Info.plist`).
  Progress is a rough estimate (`elapsed transcribed audio / total duration`,
  from Speech's partial results) — it can jump around rather than climb
  smoothly, since Speech doesn't report granular progress.

**Permission caveat.** `Info.plist` (at the package root) carries
`NSSpeechRecognitionUsageDescription`, and `Package.swift` embeds it into the
built binary via linker flags (`-sectcreate __TEXT __info_plist`) — see the
comments in both files. This is the standard workaround for a plain SPM
executable (no `.app` bundle) that needs a TCC-gated permission; without it,
macOS silently kills the process the moment `SFSpeechRecognizer.requestAuthorization`
is called, instead of prompting. If you convert this project into a proper
Xcode "App" target later, use its native Info.plist instead and remove the
linker flags.

## Requirements

- A Mac with an Apple Silicon GPU (MetalFX requires an Apple-family GPU; not
  available on Intel Macs — and macOS 27 drops Intel support entirely anyway).
- **macOS 26 "Tahoe" or later** on the Mac actually running the app (Frame
  Interpolation's hard floor).
- Xcode 26 or later (the macOS 26 SDK ships with Xcode 26), to build against it.
- `Package.swift` sets `.macOS(.v26)` as the deployment target.
- **libmpv** for playback: `brew install mpv`. (Optional: `brew install ffmpeg`
  adds subtitle generation for containers Apple's Speech framework can't
  read directly, like MKV — only the audio track is extracted, temporarily.)
- An internet connection only if the AI Subtitle Generator falls back to
  server-based recognition for a language without on-device support;
  otherwise everything in this app (upscaling, interpolation, subtitles)
  runs fully offline.

## Build & run

**Option A — Xcode:**
1. Open `Package.swift` directly in Xcode (File > Open, select the file, or
   just double-click it — Xcode treats a `Package.swift` as a project).
2. Select the `SuperResVideoPlayer` scheme and hit Run.

**Option B — command line:**
```
cd SuperResVideoPlayer
swift build -c release
swift run
```

On first launch, click **Open Video…** and pick a local `.mp4`/`.mov` file.
Toggle **Super Resolution** on/off to compare native vs. upscaled rendering,
use the segmented control to change the scale factor, use the **AI Frame
Interpolation** picker to try Off / 2x / 3x smoothing, and click **Generate
Subtitles** to transcribe the video (grant the Speech Recognition permission
prompt when it appears) — then toggle **Subtitles** or click **Export .srt…**.

## Notes & known limitations

- This is a Swift Package executable, not an `.xcodeproj` app bundle, so it
  has no true app-bundle entitlements/sandbox — `Info.plist` is embedded via
  linker flags just to satisfy the Speech framework's permission requirement
  (see the AI Subtitle Generator section above). If you later want to
  distribute this as a signed, sandboxed `.app` (e.g. for notarization), wrap
  it in an Xcode "App" target and move these source files in, using its
  native Info.plist/entitlements instead of the linker-flag workaround.
- Audio plays through the same `AVPlayer` and is not resampled or
  time-stretched, so it stays in sync with true playback time regardless of
  the interpolation multiplier — only the *video* frames shown between real
  decoded frames are synthesized for visual smoothness. This is the same
  trade-off TV "motion smoothing" features make.
- The renderer falls back to native-resolution rendering (no upscaling) if
  `MTLFXSpatialScalerDescriptor.supportsDevice(_:)` returns false for the
  current GPU, and similarly falls back to the custom warp kernel (or to
  showing the nearest real frame) if `MTLFXFrameInterpolator` is unsupported
  or not yet ready — so the app still runs on unsupported hardware, it just
  won't upscale/smooth. Both cases now surface a note in the UI
  (`PlayerViewModel.superResolutionUnsupported` /
  `.nativeFrameInterpolationUnsupported`, set by `Renderer.reportSupportState()`)
  instead of the toggle silently doing nothing.
- MetalFX's *temporal* scaler (spatial upscaling + jitter-based supersampling,
  similar to DLSS/TAA) still isn't used here for Super Resolution, since it
  also expects engine-provided motion vectors; the spatial scaler remains the
  right choice for single-frame upscaling. Frame Interpolation is the
  separate MetalFX effect this update adds, for temporal smoothing.
- Optical flow (`OpticalFlowEstimator`) runs on the CPU/ANE via Vision, off
  the render thread, at whatever cadence it can keep up with — under fast
  motion or on lower-end Macs it may lag behind real-time, in which case the
  renderer simply shows real frames without interpolation until it catches up.
  It only runs at all while Frame Interpolation is set to 2x/3x — leaving it
  Off skips optical flow entirely rather than computing it for no benefit.
  Access to its result is lock-guarded (`OpticalFlowEstimator.stateLock`)
  since it's written from a background queue and read from `Renderer.draw(in:)`,
  which MTKView calls on its own render thread, not necessarily main.
- Network/streaming URL support and drag-and-drop were left out per your
  scope (local files only) — both would be small additions to
  `PlayerViewModel.load(url:)` and `ContentView`, if you want them later.
- Subtitle generation transcribes the whole file in one pass (no chunking),
  so very long videos will take a while and use noticeable CPU/ANE; there's
  no pause/resume, only cancel-and-restart (`cancelSubtitleGeneration()`).
  Cancelling resumes the awaiting async task directly as a safety net
  (`SubtitleGenerator.cancel()`), rather than assuming
  `SFSpeechRecognitionTask.cancel()` always re-invokes the result handler.
- The cue-grouping heuristic in `SubtitleGenerator.buildCues` is intentionally
  simple (pause/length-based, not linguistic) — for high-quality subtitles
  you'd eventually want sentence-boundary detection (e.g. `NLTokenizer`) or
  manual cue editing, neither of which this project adds.

## File map

```
Package.swift
Info.plist                       — NSSpeechRecognitionUsageDescription (embedded via linker flags)
Sources/SuperResVideoPlayer/
  SuperResVideoPlayerApp.swift   — @main app entry
  ContentView.swift              — SwiftUI window, controls, subtitle overlay
  PlayerViewModel.swift          — AVPlayer wrapper, file loading, transport, subtitle state/actions
  MetalVideoView.swift           — NSViewRepresentable bridging MTKView
  Renderer.swift                 — frame history, interpolation routing, SR upscaling, draw loop
  FrameInterpolator.swift        — native MetalFX MTLFXFrameInterpolator wrapper (2x)
  OpticalFlow.swift              — Vision-based optical flow -> Metal motion texture
  Shaders.metal                  — textured triangle shaders + clear-depth/warp-blend compute kernels
  SubtitleGenerator.swift        — Speech framework transcription -> subtitle cues
  SubtitleCue.swift              — cue model + .srt export formatting
```
